## B5 — In-Place vs Root Motion

**正文**:

```
两种动画模式

In-Place：角色原地动，位移由代码驱动
Root Motion：角色带位移，动画驱动移动

Locomotion 通常用 In-Place
  + 位移完全可控
  + 方便叠加 Distance Matching 等技术

注意：资源制作时尽量保留 Root Motion 数据
  UE 导入时可提取为曲线
  Unity 根据项目需求决定
```

**配图**: 左右对比GIF — 左In-Place角色原地踏步，右Root Motion角色前移

**备注**:
> 这个区分很重要，后面Distance Matching和Stop都依赖In-Place。简单说清：In-Place是跑步机上跑，Root Motion是在地上跑。补充：制作动画时即使最终用In-Place，也尽量保留Root Motion数据——UE可以从中提取位移曲线，后面Distance Matching要用。这是美术和程序协作的关键交接点。

## B6 — 流行方案：4方向 + 骨骼扭曲

**正文**:

```
UE 当前流行的做法：少即是多

只做 4 个方向：前 · 后 · 左 · 右
中间 45° 缺口 → Orientation Warping 补

原理：
  骨盆以下朝移动方向扭曲
  上半身保持朝前
  扭曲角度可控（通常限制在 ±45° 内）
```

**配图**: 4方向分布图 + 角色不同移动方向的截图，展示骨骼扭曲范围

**备注**:
> "现在UE社区比较流行的做法——只做4个方向动画，中间角度靠骨骼扭曲补。"这是第一个重要的设计取舍：用更少的动画资源换更小的包体，代价是扭曲角度大时可能不自然。展示不同移动角度的角色截图，让观众直观看到扭曲效果。

## B12 — Loop的滑步问题

**正文**:

```
In-Place 的代价：滑步

动画有自己的步幅节奏
代码按速度驱动物体移动
速度 ≠ 动画原生速度 → 脚在地面打滑

慢速时：脚走太快，身体走太慢
快速时：脚走太慢，身体走太快
```

**配图**: GIF动图 — 不同速度下的滑步表现，慢放特写

**备注**:
> "方向问题解决了，但还有一个更根本的问题——滑步。"展示慢速和快速两种情况的滑步特写，慢放让观众看清楚脚在地面打滑。这是In-Place方案的核心代价，后面两个解法都是为了解决它。

## B13 — 解法一：修改播放速率

**正文**:

```
最直觉的修法：Play Rate = 实际速度 / 动画原生速度

优点：简单直接，一行代码
缺点：
  速度偏差大时动画加速/减速明显
  跑步变成快进走路，不自然
  本质是用时间换空间，治标不治本
```

**配图**: GIF对比 — 修Rate前 vs 修Rate后，标出速度偏差大时的不自然感

**备注**:
> "最简单的修法——把动画播放速度调成跟移动速度匹配。"先肯定它解决了大部分问题，然后指出局限：速度偏差大的时候，动画会明显变快或变慢，像快进/慢放。这是过渡到Stride Warping的动机。

---

## B14 — 解法二：Stride Warping

**正文**:

```
Stride Warping：调整步幅而非速率

原理：
  保持动画播放节奏不变
  通过缩放腿部骨骼调整步幅长度
  让步幅匹配移动速度

vs Play Rate：
  Play Rate 改时间 → 动画快进/慢放
  Stride Warping 改空间 → 步子迈大/迈小
```

**配图**: GIF对比 — 同一速度下 Play Rate vs Stride Warping，圈出步幅差异

**备注**:
> "更好的方案——不改播放速度，改步幅。"Stride Warping让角色"迈大步"或"迈小步"来匹配速度，而不是"走快点"或"走慢点"。这是UE Animation Locomotion Library提供的节点。展示效果对比：同一个高速移动，Play Rate下动画像快进，Stride Warping下步子自然变大。呼应B5："还记得我们说保留Root Motion数据吗？Stride Warping也依赖这些数据来计算步幅。"
